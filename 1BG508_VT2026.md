# 0: Knowing your way around and using Plink 

[PLINK](https://www.cog-genomics.org/plink2) is a software for fast and efficient filtering, merging, editing of large SNP datasets, and exporting to different outputs directly usable in other programs. It can store huge datasets in compact binary format from which it reads data very quickly.

It is already pre-installed on the Pelle server, you just have to load it:

```ruby
module spider plink                          ## check which PLINK versions are available
module load PLINK/2.0.0-a.6.20-gfbf-2024a    ## load the correct PLINK version
```

This is what a basic command structure looks like in `PLINK`: 

```ruby
plink --file_type_flag {INPUT_FILE} --command_flag {COMMAND_SPECIFICATION} --output_file_command --out {OUTPUT_FILE}
```

For example, to do filtering of missing markers at a cutoff of 10% missingness:

```ruby
plink --bfile file1 --geno 0.1 --recode --out file2
```

*What is the command above doing?* Reading in a bed format file (`--bfile`) called 'file1', doing the filtering (`--geno 0.1`), converting it to ped format file (`--recode`) and giving it to you (`--out`) as a file called 'file2'.


## PLINK file formats

A single data set that we could have had in a single VCF file is coded into two or three different files in the `PLINK` format. At least two files are used, one containing information about the 'family' (the samples in your data set), and one or two more files containing information about the genotypes (the alleles that each individual has, and information about those sites in the genome). These are the three 'styles' in which we can have `PLINK` files:

**ped format**: usual format (.ped and .fam)

- `.ped` - marker info
- `.fam` - sample and genotype info

**bed format**: binary/compact ped format (.fam, .bim and .bed)

- `.fam` - sample info 
- `.bim` - marker info  
- `.bed` - genotype info (binary format)

**tped format**: transposed ped format (.tfam and .tped)

- `.tfam` - sample info
- `.tped` - marker and genotype info in transposed format

# 1: Merging, Filtering, Quality control

Our analyses can very rarely be run on raw data. We often apply several filters to remove either genetic markers/sites or individuals of poor quality, although there are no perfect thresholds for these filters. We usually decide based on tradition (so we can compare our results to previous literature) or a little bit of testing (to see what works best for our data).

First, we filter for **marker** missingness. Paste in the command below to filter out markers with more than 10% missingness:

## Step 3 - MAF filtering

Filtering sites based on minimum allele frequency (MAF) is not always optimal, especially if you are going to merge your data with other datasets in which the low-frequency alleles might be present. We will apply a MAF filter now to showcase its use, but in real research we would apply it when all data has been merged:

Filter data for a MAF of 1%:

```ruby

```

*How many SNPs are left in the data set at this point?*  

## Step 4 - Filtering for SNPs out of HWE

SNPs that are not in Hardy-Weinberg equilibrium (HWE) usually indicate problems with the genotyping/sequencing. However, to avoid filtering out SNPs that are selected for/against in certain groups (especially when working with case/control data) filtering HWE *within groups* is recommended. Now, we will just filter the entire data set for SNPs that aren’t in HWE with a significance threshold of 0.001. Again, in real research, this filter is better applied when all data has been merged.

```ruby
plink --bfile unk4 --hwe 0.001 --make-bed --out unk5
```

*How many SNPs were excluded?*  

If you want to look at the results from the HWE test, you can use the command below. By running it, you can also obtain the observed and expected heterozygosities at each site:

```ruby
plink --bfile unk5 --hardy --out hardy_unk5
```

Look at file 'hardy_unk5.hwe' (with `head` or `less`), do you understand the output?

The output file has the following format:

- `SNP`: SNP identifier
- `TEST`: Code indicating sample
- `A1`: Minor allele code
- `A2`: Major allele code
- `GENO`: Genotype counts: AA/AB/BB 
- `O(HET)`: Observed heterozygosity
- `E(HET)`: Expected heterozygosity
- `P`: H-W p-value (Chi-squared test)

There are additional filtering steps that you can do. `PLINK` website lists all the cool commands that you can use to treat your data. Usually, we also filter for related individuals and do a sex-check on the X-chromosome to check for sample mix-ups. 

## Step 5 - Filtering out related individuals (OPTIONAL)

In most genomic analyses, we don't want any related individuals: they're basically giving us gene copies of the same origin, which violate assumptions of independent sampling, and they don't add valuable information compared to a new, unrelated individual. In order to identify related pairs of individuals, we usually use [KING](http://people.virginia.edu/~wc9c/KING/manual.html). Feel free to check out the manual and figure out how the software works!  

The input files for `KING` need to be in `PLINK` binary format, i.e. `.bed`, `.fam`, and `.bim` files. A binary format allows efficient compression of genotype data by using two bits to represent a genotype, which offers substantial computational savings that are essential to `KING` analysis.  

Before starting, we need to change the last column of the `.fam` file, because `KING` does not like some numbers, and all individuals in the data set should have the same 'family names' (otherwise, `KING` will only look for relatives within each family):  

```ruby
# Change last column to "1", not problematic
DB=unk5
awk '{print $1, $2, $3, $4, $5, 1}' ${DB}.fam > ${DB}.fam1
mv ${DB}.fam1 ${DB}.fam

# Change family names to "COMMON_FAM" in .fam file
DB=unk5
cp ${DB}.fam ${DB}.fam_original
awk '{print "COMMON_FAM", $2, $3, $4, $5, $6}' ${DB}.fam_original > ${DB}.fam
```

Now, we can actually run `KING`. We will start by making a directory where all results will be put:

```ruby
# Create KING directory where results will be and remove any previous results with the same data set
# You should be where the PLINK data set is, do not enter this directory yet
mkdir -p KING
rm KING/${DB}_king_info.txt

# Write related individuals
# Check output: KING/{...}_kinship_list.txt
DB=unk5
/proj/uppmax2026-1-105/private/king -b ${DB}.bed --kinship --prefix KING/${DB} > KING/${DB}_king_info.txt
tail -n20 KING/${DB}_king_info.txt
awk '$1=$1' KING/${DB}.kin > remove_file.txt
sed 's/ /\t/g' remove_file.txt > KING/${DB}.kin
rm remove_file.txt
wc -l KING/${DB}.kin
sort -k9 -r -h KING/${DB}.kin > KING/${DB}_kinship_list.txt
ll KING; echo ""
head KING/${DB}_kinship_list.txt
```

*What do you see in the output of the previous commands? How many related pairs are there? To what degree are they related?*

If we look into the 'kinship_lists.txt' file, we can count whether it's better to remove individuals from the first or second column, because we want to remove our minimal number of data points:

```ruby
number_of_related_pairs=2

# First do 'wc -l' with both 'cut -f3' and 'cut -f2' to see which one is lower:
head -n ${number_of_related_pairs} KING/${DB}_kinship_list.txt | cut -f3 | sort | uniq | wc -l ## count inds in col 3: 27
head -n ${number_of_related_pairs} KING/${DB}_kinship_list.txt | cut -f2 | sort | uniq | wc -l ## count inds in col 2: 27

# Individuals are sorted by kinship coeff (9th column), so you can choose the sum of "FS" and "2nd" degree relatives
head -n ${number_of_related_pairs} KING/${DB}_kinship_list.txt | cut -f3 | sort | uniq > ID_list.txt
```

Finally, we can remove those related individuals from the initial data set, but we first need to give all original family names back:

```ruby
# Replace .fam with .fam_original
DB=unk5
mv ${DB}.fam_original ${DB}.fam
```

## Step 7 - Data Merging & strand flipping (OPTIONAL)

The next step would be to start merging your data with comparative data sets. Usually, when you merge your data with another data set, there are strand issues. The SNPs in the other data set might be typed on the reverse DNA strand and yours on the forward, or viceversa. Therefore, you need to flip the strand of one of the data sets for all the SNPs that show a strand mismatch. We should not flip `C/G` and `A/T` SNPs because we cannot distinguish reverse and forward orientation (i.e. `C/G` becomes `G/C` unlike other SNPs i.e. `G/T` which become `C/A`). Therefore before merging and flipping, all `A/T` and `C/G` SNPs must be excluded. However, this can be a problem since some of your SNPs in your data set may be monomorphic, i.e. in the `bim` file they will appear as `C 0` (with `0` meaning missing). So you don't know what kind of SNP it is, it can be `C/G`, `C/T` or `C/A`. If it is `C/G`, it needs to be filtered out, but that's not needed if it is `C/T`.

Therefore, before merging our data with other data sets, it is important to merge your data with a fake heterozygous 'reference individual': our `RefInd1`. This is an individual that is heterozygous at every SNP position. This 'fake' genome you can prepare from the SNP info file you get from the genotyping company or your own genomics processing software (such as Genome Studio from Illumina). You can also prepare it from data downloaded for each SNP from a web-database such as dbSNP. 

We will start by extracting our SNPs of interest from the 'RefInd' (remember that you filtered out several SNPs in previous steps, so the intiial files 'unk1' contain more SNPs than the ones we actually want).

```ruby
plink --bfile RefInd1 --extract unk5.bim --make-bed --out RefInd1_ext 
```

The code below makes a list of `C/G` and `A/T` SNPs in your data set (or reference individual, they contain the same SNPs at this point):

```ruby
sed 's/\t/ /g' RefInd1_ext.bim | grep " C G" > ATCGlist
sed 's/\t/ /g' RefInd1_ext.bim | grep " G C" >> ATCGlist
sed 's/\t/ /g' RefInd1_ext.bim | grep " A T" >> ATCGlist
sed 's/\t/ /g' RefInd1_ext.bim | grep " T A" >> ATCGlist
```

Exclude the `C/G` and `A/T` SNPs form both your reference individual and data set:

```ruby
plink  --bfile RefInd1_ext --exclude ATCGlist --make-bed --out RefInd1_ext2 
plink  --bfile unk5 --exclude ATCGlist --make-bed --out unk6
```

Merge your data set 'unk6' with 'RefInd_ext2':

```ruby
plink --bfile RefInd1_ext2 --bmerge unk6.bed unk6.bim unk6.fam --make-bed --out MergeRef1  
```

An error is generated because of the strand mismatches. The generated file 'MergeRef1.missnp' contains the info on the SNPs that show mismatches. In the `PLINK` output message, you are advised to flip the strand of these SNPs in your data set.

```ruby
plink --bfile unk6 --flip MergeRef1-merge.missnp --make-bed --out unk7  
```

Now you can try the merging again, between the reference individual (unaltered) and your data set (with some SNPs flipped):

```ruby
plink --bfile RefInd1_ext2 --bmerge unk7.bed unk7.bim unk7.fam --make-bed --out MergeRef2  
```

It should have worked now, right?

Now that we have our data set together with a fake, all-heterozygous individual, we can actually merge our data with a set of reference populations that we downloaded from published studies (e.g. KGP, HGDP, SGDP). Many of the sites archiving the data provid them in `PLINK` format. For this practical, we selected a few reference populations from KGP and HGDP to use as comparison with your unknown populations.

Look at the 'refpops1.fam' file (with `head` or `less`). *Do you recognize some of these population names (you can do a quick online search)?* There are two HapMap and three HGDP populations.

## Step 8 - LD Pruning

Before running PCA and ADMIXTURE, it is recommended to prune the data a bit in order to thin the marker set for linkage disequilibrium (LD). In a nutshell, we use the first SNP (in genome order) and compute its correlation with a few following SNPs (e.g. the next 50). When we find a large correlation, we remove one SNP from the correlated pair, keeping the one with the largest minor allele frequency (MAF), thus possibly removing the first SNP. Then we go on with the next SNP not yet removed. You can read up on how `PLINK` prunes for LD [here](https://dalexander.github.io/admixture/admixture-manual.pdf).

In `PLINK`, we first test which SNPs are in LD:

```ruby
plink --bfile MergeRefPop3 --indep-pairwise 10 10 0.1
```

And then we remove them (by default, SNPs to keep are written in 'plink.prune.in' and SNPs to remove are written in 'plink.prune.out'):

```ruby
plink --bfile MergeRefPop3 --extract plink.prune.in --make-bed --out MergeRefPop3_LDpruned
```

You have now obtained the files you need to actually run the interesting analyses (PCA and ADMIXTURE).

## Step 2 - Plot the PCA in R

The interesting output files from the PCA analysis end in `eigenval` and `eigenvec`.

In the `.eigenvec` file is the main output for the number of principal components that you specified (default of 20). The first and second column specify the family (population) and individual (sample) names for each of your samples. The res of the columns contain the eigenvectors from each of the 20 computed principal components. The `.eigenval` lists the 20 eigenvalues, which describe the proportion of variance each of the principal components is summarizing. To work out the percentage of variation each PC explains, you divide your particular PC eigenvalue by the sum over all the eigenvalues.

## Step 3 - ADMIXTURE 

`ADMIXTURE` is a clustering software that we use with the aim to infer population and individual ancestries. It is nevertheless a clustering algorithm, which only describes the data at hand, like PCA, and does not test any specific hypothesis. In broad strokes, it splits the genome into smaller segments, and asks the question "Which ancestry does this segment of the genome match best?". The number of 'ancestries' we specify ourselves (e.g. 2, 3, 4, 5), and `ADMIXTURE` splits the genetic data into that many group, by maximizing within-group similarity (in terms of allele frequencies).

`ADMIXTURE` runs directly from `.bed` or `.ped` files and needs no extra parameters for file preparation. 

A basic `ADMIXTURE` run looks like this, where 'K' is the number of clusters you want to split the data into:

```ruby
admixture -s time {your_dataset}.bed K
```

We need to run many iterations at each K value, because there can be some stochasticity in the clustering process. Therefore, a compute cluster and some scripting are very useful.

First, we have to check what `ADMIXTURE` is called in Pelle:

```ruby
module spider ADMIXTURE
module load ADMIXTURE/1.3.0
```

Create a script file (using `nano`, for example: `nano run_admixture.sh`) and paste in the code below, which will run ADMIXTURE for five values of K (2 to 6) and 3 iterations at each K value.

```ruby
#!/bin/bash

# loop over K values
for kval in {2..6}; do

   # loop over the iterations
   for itir in {1..3}; do
           (echo '#!/bin/bash -l'
           echo "

           ## create a folder with the K value and iteration value
           mkdir ${kval}.${itir}

           ## move into this folder
           cd ${kval}.${itir}

           ## load and run ADMIXTURE for that specified K
           module load ADMIXTURE/1.3.0
           admixture -j3 -s $RANDOM $1 ${kval}

           ## move one folder up
           cd ../
           exit 0") |

           ## send all the previous code as an independent job
               sbatch -n 3 -t 72:00:00 -A uppmax2026-1-105 -J ADMX.${kval}.${itir} -o ADMX.${kval}.${itir}.output -e ADMX.${kval}.${itir}.message

       ## continue with the next iteration on the same K
       done

## continue with the following K
done
```

You can see a `$1` in the script. This is asking for an input file to be given to the `run_admixture.sh` script, which will be the *full directory* and *file name* of your data set. In my case, sending the job will look like this:

```ruby
YOUR_NAME='andrea'
bash run_admixture.sh /proj/uppmax2026-1-105/private/Population_structure_lab/${YOUR_NAME}/DATA/MergeRefPop3_LDpruned.bed
```

After a successful `ADMIXTURE` run, you should be seeing a bunch of new folders being created in the ADMX directory, as well as some 'output' and 'message' files. In each of the folders, you will find the output `.P` and `.Q` for each iteration. These are the files we're interested in, but the 'message' file should inform you about any errors in your script.
